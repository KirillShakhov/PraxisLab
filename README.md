# DecentralizedVLab

Прототип Local-First платформы виртуальных лабораторий с клиентским исполнением Python, JavaScript, Lua и SQLite, многофайловым редактором, CRDT-синхронизацией через SignalR, локальным хранением курсов и офлайн-режимом.

## Запуск через Docker

Основной способ запуска проекта:

```powershell
docker compose up -d --build
```

После запуска приложение доступно по адресу:

```text
http://localhost:8080
```

Состав стенда:

- `gateway` — YARP reverse proxy, внешний порт `8080`;
- `frontend` — production-сборка React/Vite, отдаётся через nginx;
- `backend` — ASP.NET Core API, SignalR Hub, REST-синхронизация курсов и прокси к сервису исполнения;
- `execution` — отдельный ASP.NET Core сервис запуска учебного кода для серверного режима сравнения профилировщика.

Проверить состояние контейнеров:

```powershell
docker compose ps
```

Остановить стенд:

```powershell
docker compose down
```

## Локальная разработка фронтенда

Для изменения клиентской части можно запускать фронтенд отдельно:

```powershell
cd frontend
npm install
npm run dev
npm run lint
npm run build
```

Backend и gateway собираются основным Docker-сценарием из секции выше. Dockerfile-ы используют окружение .NET 10, поэтому отдельная локальная установка .NET SDK для запуска стенда не требуется.

## Воспроизведение benchmark

1. Запустить стенд через `docker compose up -d --build`.
2. Открыть `http://localhost:8080`.
3. Создать или открыть лабораторную работу на Python, JavaScript, Lua или SQLite.
4. Открыть панель профилировщика.
5. Для клиентского сценария выполнить 200 итераций после прогрева.
6. Для сравнения с сервером включить режим сравнения: запросы идут через gateway в `/api/execute`, backend передаёт их в отдельный `execution`-контейнер, где запускается соответствующий интерпретатор.
7. Экспортировать отчёт профилировщика в Markdown.

Важно: серверный benchmark в текущей постановке является благоприятным для серверной модели. Код выполняется в отдельном постоянном `execution`-контейнере Docker Compose, но стенд не создаёт новый контейнер, gVisor-песочницу или microVM на каждый запуск пользовательского кода.

## Офлайн-режим

Офлайн-работа доступна после первичной загрузки приложения и используемых компиляторов. Первая сессия требует сеть для установки Service Worker и загрузки WebAssembly-ресурсов. После кэширования доступны локальное редактирование, сохранённые курсы и запуск закэшированных сред исполнения; совместная работа и REST-синхронизация требуют восстановления сети.
